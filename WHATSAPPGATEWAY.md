# WhatsApp Gateway Service Documentation

## 1. Introduction

This document describes a WhatsApp Gateway service built with Go using the [`go-whatsapp-web-multidevice`](https://github.com/aldinokemal/go-whatsapp-web-multidevice) library. The service exposes a REST API for sending WhatsApp messages and supports multi-device operation, allowing each request to target a specific connected WhatsApp account through device scoping.

The gateway is suitable for backend integrations, internal tools, notification services, and automation workflows that require programmatic message delivery over WhatsApp.

## 2. Installation

### Prerequisites

Before running the service, ensure following dependencies are available:

- Go 1.24 or newer
- FFmpeg
- libwebp / WebP utilities
- Git

> On Windows, WSL is generally recommended by upstream project for smoother dependency handling.

### Clone Repository

```bash
git clone https://github.com/aldinokemal/go-whatsapp-web-multidevice.git
cd go-whatsapp-web-multidevice/src
```

### Install Dependencies

Go will resolve dependencies automatically when project is built or executed.

```bash
go mod download
```

### Prepare Environment File

Copy example environment file and adjust values as needed:

```bash
cp .env.example .env
```

If `cp` is not available on Windows PowerShell, use:

```powershell
Copy-Item .env.example .env
```

### Run Service in REST Mode

```bash
go run . rest
```

Service will start in REST mode and, by default, listens on port `3000` unless overridden by configuration.

### Optional: Build Binary

```bash
go build -o whatsapp
./whatsapp rest
```

On Windows PowerShell:

```powershell
go build -o whatsapp.exe
.\whatsapp.exe rest
```

## 3. Configuration

This service supports configuration through command-line flags, environment variables, and `.env` file. For credential management, recommended approach is to store API authentication values in `.env`.

### Basic Authentication Credentials

Set gateway credentials in `.env` using `APP_BASIC_AUTH`.

```env
APP_BASIC_AUTH=your_username:your_password
APP_PORT=3000
APP_HOST=0.0.0.0
APP_DEBUG=false
```

Example:

```env
APP_BASIC_AUTH=example:example
APP_PORT=3000
APP_HOST=0.0.0.0
APP_DEBUG=true
```

### Explanation

- `APP_BASIC_AUTH` defines HTTP Basic Auth credentials accepted by API.
- Format is `username:password`.
- Service also supports multiple credential pairs separated by commas, for example:

```env
APP_BASIC_AUTH=admin:supersecretpassword,service-user:anothersecret
```

### Configuration Priority

Configuration priority follows upstream behavior:

1. Command-line flags
2. Environment variables
3. `.env` file

If same value is defined in more than one place, higher-priority source overrides lower-priority source.

### Security Recommendations

- Do not hardcode username or password in source code.
- Do not commit `.env` into version control.
- Use strong passwords for `APP_BASIC_AUTH`.
- Place service behind firewall, reverse proxy, or private network when possible.
- Use HTTPS in production, especially when Basic Auth credentials are sent over network.

## 4. API Reference

### Base URL Format

Base URL follows this structure:

```text
http://username:password@IP_address:PORT
```

Example:

```text
http://admin:supersecretpassword@127.0.0.1:3000
```

> Embedding credentials in URL is useful for quick testing. In production integrations, prefer sending credentials through standard `Authorization: Basic ...` header generated by HTTP client.

### Authentication

API uses HTTP Basic Authentication.

Credentials are validated against values configured in environment via `APP_BASIC_AUTH`.

Mechanism summary:

- Client sends username and password using Basic Auth.
- Server validates credentials against configured auth pairs.
- If credentials are missing or invalid, request is rejected.

### Endpoint

```http
POST /send/message
```

### Required Headers

| Header | Required | Description |
| --- | --- | --- |
| `Authorization` | Yes | HTTP Basic Auth header generated from configured username and password. |
| `X-Device-Id` | Yes* | Device identifier for target WhatsApp account in multi-device setup. |
| `Content-Type` | Yes | Must be `application/json`. |

> `X-Device-Id` is required for device-scoped requests in multi-device environments. If only one device is registered, upstream service may use it as default, but explicit header is recommended for predictable behavior.

### Request Body

```json
{
  "phone": "6281310307754@s.whatsapp.net",
  "message": "sample message",
  "reply_message_id": "",
  "is_forwarded": false,
  "action": "stop",
  "duration": 86400
}
```

### Field Descriptions

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `phone` | `string` | Yes | WhatsApp JID of message recipient. For direct chat, use format like `6281310307754@s.whatsapp.net`. |
| `message` | `string` | Yes | Text message content to send. |
| `reply_message_id` | `string` | No | Message ID being replied to. Leave empty string if request is not a reply. |
| `is_forwarded` | `boolean` | No | Marks outgoing message as forwarded when set to `true`. Default behavior is `false`. |
| `action` | `string` | No | Disappearing-message behavior or related message action handled by upstream endpoint. Value `stop` commonly indicates disabling timed disappearing action for message context. Use according to service behavior in your deployment. |
| `duration` | `integer` | No | Duration in seconds associated with action context. Example `86400` equals 24 hours. Useful when endpoint behavior supports timed message state or disappearing-message duration. |

### Notes for `phone` Format

Common JID formats:

- Personal chat: `6281310307754@s.whatsapp.net`
- Group chat: `<group_id>@g.us`

Always send recipient identifier in full WhatsApp JID format expected by service.

## 5. Usage Example

Below is complete example using `curl` with Basic Auth, JSON payload, and required `X-Device-Id` header.

```bash
curl --request POST "http://127.0.0.1:3000/send/message" \
  --user "admin:supersecretpassword" \
  --header "Content-Type: application/json" \
  --header "X-Device-Id: 628123456789@s.whatsapp.net" \
  --data '{
    "phone": "6281310307754@s.whatsapp.net",
    "message": "sample message",
    "reply_message_id": "",
    "is_forwarded": false,
    "action": "stop",
    "duration": 86400
  }'
```

### Equivalent Request Using Base URL Credential Format

```bash
curl --request POST "http://admin:supersecretpassword@127.0.0.1:3000/send/message" \
  --header "Content-Type: application/json" \
  --header "X-Device-Id: 628123456789@s.whatsapp.net" \
  --data '{
    "phone": "6281310307754@s.whatsapp.net",
    "message": "sample message",
    "reply_message_id": "",
    "is_forwarded": false,
    "action": "stop",
    "duration": 86400
  }'
```

## Additional Operational Notes

- Run service in REST mode using `rest` command.
- Ensure at least one WhatsApp device is connected and authenticated before sending messages.
- In multi-device deployments, always provide `X-Device-Id` to avoid routing message through wrong account.
- For production deployment, use HTTPS and external secret management where possible.
